# Welcome!

Thank you for taking part in this experiment! The goal of this study is to explore how different development approaches affect the outcome of LLM-generated code.

You are part of **Group A**. Your task is to build a time-logging web application with authentication, role-based access, and admin functionality. You'll be following a modified Test-Driven Development (TDD) process — this means you'll write tests by hand first and then ask the LLM to generate the code that makes those tests pass, following a Red, Green, Blue cycle.

Everything you need to get started is described below. Please read through the user stories and instructions carefully before you begin, and don't forget to run the provided bash scripts after each phase so we can track your progress. Good luck, and have fun!

## The process
**Note**:After each phase you need to run the provided bash file for the phase(ex. if you just completed red phase you run red.sh) so that we can log you are following the process correctly. **Please note that if you do not do this we would have to invalidate your results**
- Red phase: You are to write tests manually (no LLM) for a functionality. After this you **need** to run the test and see it fail. In this modified version of TDD you are allowed to write more than one test at a time, but it NEEDS to cover the same feature, you cant write multiple tests for different features at the same time
- Green phase: ONLY after completing the red phase you can begin this phase. You are to ask the LLM to implement the minimal code needed to pass the test you just wrote manually.If the tests still fail, ask the LLM to fix the implementation. Repeat until green. After that you need to run the test and see it pass.
- Blue phase (Refactor):  ONLY after completing the green phase you can begin this phase. You are to refactor the code so that it is clean and improved. You are **NOT ALLOWED** to add new functionality.
- Repeat!

# Authentication
## Requirement: Users must authenticate to reach protected routes.
#### Role-Based Access Control Logic:
- Potential User(Not logged in): Access to `/login`, `/register`. Redirected to `/user` if already logged in.
- USER(Logged in): Access to `/user`, `/logout`. Unauthorized for `/admin`. Redirect away from `/login` and `/register`.
- ADMIN(Logged in): Access to `/user`, `/admin`, `/logout`. Redirect away from `/login` and `/register`.

# Application structure
## `src/main/java/com.experiment/`
This is where the code is located, and it is further divided into these packages:
### `/controller`
All files inside the controller handles the API calls, this is where you can implement the logic for the new API calls and edit them.
At the moment there are some boilerplate data inside these files just so that we can showcase it in the frontend. You can remove those if you want.
### `/model`
This is where the base classes (Object classes) are located. They have everything necessary in them but if you want you can modify them.
There are two helper functions inside the User class (stringifyUser,calculateTotalLoggedTime) that are there to make sure that the frontend gets the correct JSON structure. try to not modify these or the fron-end might break.
### `/repository`
This is where the database managers are located, here you can for example write a function to add a User and so on.
It is the middleman between the API and the Sqlite database.
### `/service`
Business logic lives here, sitting between controllers and repositories.
### `App.java`
This is where the Application and classes are initialized. The routes are defined here as well, you can modify as you like.
### `Database.java`
The database have every necessary table and variables but if you want to modify it, go ahead.
## `src/main/java/resources/`
The HTML files are located here, they should not be modified, but you can if you really want to.
## `src/test/java/com.experiment/`
The tests are located here, there is some boilerplate to get you started. You can modify as you please.
